같은 조건에서 Codex와 Claude Code의 수행 방식 차이.


당연한 얘기 같지만 조건을 잘 통제한 환경에서 비교를 한다면 모델의 성향에 따라서 결과가 달라진다.


여기서 부터 Claude Code(Opus 4.6)의 의견.

1. 모델의 "코딩 성향" 차이
각 모델이 학습 과정에서 내재화한 "좋은 코드"의 기준이 다릅니다.
이건 "어느 쪽이 맞다"가 아니라 학습 데이터에서 어떤 코드를 더 많이, 더 높은 가중치로 봤느냐의 차이입니다.


2. 의사결정 분기점의 누적
프로젝트 초반에 작은 선택 하나가 이후 전체 구현에 영향을 줍니다.
notify-001에서 user_id를 str로 정의 (Claude)
 → notify-002에서 비교 로직도 str 기반
 → notify-003에서 조회 방식도 달라짐

notify-001에서 user_id를 UUID로 정의 (Codex)
 → 이후 전체가 UUID 기반으로 일관

stock-005에서도 마찬가지:
Claude: worker 안에서 fetch 직접 처리 → retry도 수동 루프 → US도 같은 패턴
Codex: client 모듈 분리 → Celery retry 자연스럽게 적용 → US도 별도 client

첫 feature의 아키텍처 결정이 도미노처럼 이후 feature에 전파됩니다.


3. 외부 라이브러리/데이터 소스 지식 차이
가장 실질적인 차이가 여기서 나옵니다.

- US 종목 데이터: Claude는 yfinance를 선택, Codex는 NASDAQ Trader FTP를 선택
- Celery retry: Claude는 수동 루프, Codex는 self.retry() 네이티브

이건 모델이 해당 라이브러리의 best practice를 얼마나 깊이 알고 있느냐의 차이입니다. Codex가 Celery의  self.retry()를 선택한 건 OpenAI 학습 데이터에 Celery 실전 코드가 더 많았을 가능성이 있고, Claude가 yfinance를 선택한 건 Python 금융 데이터 관련 코드에서 yfinance가 자주 등장했기 때문일 겁니다.


4. 프롬프트 해석의 미세한 차이
동일한 spec을 읽어도 "무엇을 중요하게 볼 것인가"가 다릅니다.

spec: "알림 페이지에서 전체 알림 이력 조회 가능"
- Claude 해석: "조회 기능 구현" → API 래퍼 만들고, 페이지 렌더링
- Codex 해석: "조회 가능" = "최신 상태여야 함" → 패널 열 때마다 재조회, 읽음 즉시 반영

같은 문장에서 Claude는 기능 완성에 초점, Codex는 사용자 경험에 초점을 맞춘 셈입니다.


결론

차이의 본질은 "같은 요구사항을 보고도 다른 트레이드오프를 선택한다"는 것입니다.
사람 개발자 두 명에게 같은 Jira 티켓을 줘도 구현이 다른 것과 같은 원리인데, 모델의 경우 그 성향이 학습 데이터와 RLHF에 의해 결정된다는 점이 다릅니다.
