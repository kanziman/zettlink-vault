AI 사용의 복리 성장을 위한 다섯가지 개념도입니다. 구체적으론 훨씬 많은 디테일이 있지만, 중요한건 변하지 않는 핵심이겠죠. 
​
1. 한 번 하면 두 번 안 한다 — "스킬화"
패턴: AI에게 시킨 작업이 잘 되면, 그걸 버리지 말고 재사용 가능한 스킬로 만든다.
​
ChatGPT의 GPTs나 Gemini의 Gem을 떠올릴 수 있습니다. 하지만 써보신 분은 아실 겁니다 — 설정이 제한적이고, 실제 파일을 읽거나 수정할 수 없고, 코드를 실행할 수도 없어서 금방 막힙니다. "이 정도면 되겠지"에서 멈추는 도구입니다.
​
반면 Claude Code 같은 풀 자유도 환경에서 만드는 스킬은 차원이 다릅니다. 파일시스템을 읽고, 코드를 실행하고, API를 호출하고, 다른 스킬을 불러옵니다. 진짜로 모든 일을 다 합니다.
​
예를 들어 견적서 스킬은 이런 일을 합니다: 고객이 개인이면 개인용 템플릿을 열고, 기업이면 기업용 템플릿을 엽니다. 별도 파일에 정리된 최신 원가표를 참조해서 테이블을 채우고, 엑셀의 셀 서식까지 맞춥니다. 중간에 뭔가 안 맞으면 스스로 다시 생각해서 다른 방법을 시도합니다. 이 전 과정이 하나의 스킬입니다. 사용자가 하는 일은 "딸깍" — 버튼 하나 수준입니다.
​
그리고 스킬은 계속 나아집니다. 저번에는 잘 됐는데 이번에는 결과가 좀 달랐다면, "이번에 이런 문제가 있었으니 고쳐줘"라고 하면 됩니다. AI가 전 과정을 기록해서 스킬을 업그레이드합니다. 쓸수록 정교해지는 업무 매뉴얼인 셈입니다 
​
— 다만 사람이 읽는 매뉴얼이 아니라 AI가 실행하는 매뉴얼이라는 점이 다릅니다.

실제 사례:
견적서를 처음 만들 때 30분 걸렸습니다. 스킬로 만든 뒤 두 번째부터는 5분입니다.
그 견적서 스킬에서 제안서 스킬이 파생되고, 다시 계약서 스킬로 발전했습니다.
한 달 사이에 20개가 넘는 스킬이 서로 연결되어 돌아가게 됐습니다.
​
핵심: GPTs나 Gem은 "설정"입니다. 스킬은 "자동화"입니다. 설정은 한계가 있고, 자동화는 한계가 없습니다. AI를 매번 쓰는 도구가 아니라 점점 나아지는 시스템으로 만드세요. 한 번의 노력이 복리로 돌아옵니다.

2. 메시지 하나가 실제 업무가 된다 — "실행형 대화"
​
패턴: AI에게 "이거 해줘"라고 말하면, 대화로 끝나는 게 아니라 실제로 파일이 생기고 결과물이 나온다.
보통 AI 챗봇에게 "랜딩 페이지 만들어줘"라고 하면 코드를 텍스트로 보여줍니다. 복사해서 어디에 붙여넣을지는 내 몫이죠. 서버 위에서 돌아가는 AI는 다릅니다. 말하면 파일이 생기고, 바로 배포까지 됩니다. 대화창이 곧 업무 시스템입니다.

3. 내가 배운 건 팀 전체가 쓴다 — "공유 메모리"
​
패턴: 한 사람이 AI에게 가르친 것이 팀 전체의 자산이 된다.
​
지금 대부분의 회사에서 AI는 개인 도구입니다. 내가 한 대화를 동료가 볼 수 없고, 같은 질문을 5명이 각각 합니다. 누군가 훌륭한 프롬프트를 찾아도 슬랙에 공유하고 끝 — 맥락은 전달되지 않습니다.
​
공유 메모리가 있는 환경에서는 완전히 달라집니다. 한 사람이 월요일에 "우리 회사 브랜드 톤은 이렇다"고 가르치면, 금요일에 합류한 신입도 그 맥락 위에서 바로 일합니다. AI가 이미 회사를 알고 있으니까요.

4. 큰 일은 쪼개서 동시에 돌린다 — "태스크 분해"
패턴: 큰 업무를 AI에게 주면, AI가 스스로 하위 태스크로 쪼개서 각각 자율 실행한다.
​
보통 AI에게 "제품 런칭 기획해줘"라고 하면 긴 텍스트 하나가 나옵니다. 읽고 판단하고 다음 단계를 지시하는 건 결국 나의 몫입니다. 태스크 보드가 있는 환경에서는 다릅니다 — 하나의 지시가 시장 조사, 경쟁사 분석, 가격 전략, 타임라인이라는 네 개의 독립 태스크로 쪼개져서 각각 돌아갑니다.

5. 결정은 기록하고, 기록은 검색한다 — "AI 시대의 문서화"
​
패턴: AI와 함께 내린 결정을 구조화된 기록으로 남기고, 나중에 검색해서 맥락을 복원한다.
​
AI와 일하면 의사결정 속도가 빨라집니다. 하루에도 수십 가지 결정이 내려집니다. "이 가격으로 가자", "이 구조로 제안하자", "이건 다음 분기로 미루자". 문제는 속도가 빨라진 만큼, 기록하지 않으면 맥락이 순식간에 사라진다는 겁니다.
우리 팀은 결정을 날짜+제목으로 기록하고, 반드시 "왜"를 포함합니다. "A 대신 B를 선택한 이유는 X." 이렇게 쌓인 기록은 한 달 뒤 클라이언트가 "왜 이렇게 했나요?"라고 물을 때, 검색 한 번으로 근거를 제시할 수 있게 해줍니다.
