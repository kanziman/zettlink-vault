남편이 회사 개발자에게 들었다며 
RTK라는 도구를 알려줬습니다.

“Claude Code 쓸 때 토큰 줄여주는 툴이래.”

처음엔 그냥 비용 아끼는 작은 유틸리티 정도로 생각했습니다.

무지성으로 설치부터 해볼까 싶다가
요즘 AI 코딩을 좀 더 효과적으로 하고 싶기도 한터라, 이게 왜 필요한 도구인지 들여다보게 되었습니다.

—

AI 코딩에서 비용을 잡아먹는 게 
꼭 “복잡한 추론”만은 아니더라고요.

Claude Code 같은 에이전트는 작업 중에 끊임없이 터미널 명령을 실행합니다. 
git status, git diff, ls, grep, npm test, cargo test 같은 것들을 돌리죠.

사람 개발자는 이런 출력을 빠르게 훑고 핵심만 봅니다. 그런데 AI 에이전트는 기본적으로 이 출력을 텍스트로 다 읽고, 컨텍스트에 넣고, 그 위에서 판단합니다.

문제는 그 출력 대부분이 
실제 판단에 꼭 필요한 정보가 아니라는 점입니다.
사실 사람의 의사결정도 머릿속을 들여다보면 비슷하겠죠. 

반복되는 로그, 긴 파일 목록, 통과한 테스트 메시지, 이미 알고 있는 디렉토리 구조까지 통째로 들어갑니다. 사람에게는 배경 소음처럼 지나가는 정보도, AI에게는 모두 토큰입니다.

게다가 AI 에이전트는 한 번 컨텍스트에 들어간 출력을 매 턴마다 다시 읽습니다. 첫 턴에서 git diff 출력 2,000 토큰이 들어가면, 그다음 10턴 동안 그 2,000 토큰이 매번 재독됩니다. 명령 한 번이 그 자리에서만 끝나는 게 아닌 거죠.

이걸 알고 나니, 그동안 세션이 길어질수록 Claude의 판단이 조금씩 흔들리는 듯한 느낌이 왜였는지 어렴풋이 이해가 됐습니다.

—

RTK는 바로 이 지점에 끼어드는 도구입니다.

Rust Token Killer의 약자로, AI 코딩 에이전트와 터미널 사이에 앉아서 명령 출력이 컨텍스트에 들어가기 전에 한 번 걸러주는 CLI 프록시입니다.

깃허브 저장소 기준 평균 60~90% 압축률을 보고하고 있고, 한 사용자는 30분 세션에서 약 150K 토큰이 45K 토큰으로 줄었다고 측정한 결과를 공유하기도 했습니다.

작동 방식 자체는 그렇게 복잡하지 않습니다. 네 가지 전략을 순서대로 적용한다고 합니다.

ANSI 컬러 코드나 진행 바 같은 노이즈를 걷어내고(Smart Filtering), 비슷한 항목을 묶고(Grouping), 반복되는 패턴을 중복 제거하고(Deduplication), 성공 메시지는 줄이되 에러와 경고는 살려두는 방식(Truncation)입니다.

토큰을 마법처럼 줄이는 게 아니라, AI가 읽게 되는 출력을 덜 낭비적으로 만들어주는 겁니다.

—

설치는 macOS 기준 Homebrew로 가능하고, Claude Code뿐 아니라 Cursor, Gemini CLI, Codex 등에도 hook을 설치할 수 있습니다.

한 가지 주의할 점은 crates.io에 “Rust Type Kit”라는 동명 패키지가 따로 있다는 거였습니다. 

cargo install rtk로 그냥 설치하면 엉뚱한 게 깔릴 수 있어서, GitHub 저장소(rtk-ai/rtk)에서 직접 설치하거나 Homebrew를 쓰는 편이 안전합니다. 

설치 후 rtk gain 명령이 잘 작동하는지로 제대로 깔렸는지 확인할 수 있습니다.

—

다만 써보면서 느낀 건, 
이게 항상 켜두면 좋은 만능은 아니라는 점입니다.

압축이 있다는 건 필연적으로 일부 정보가 생략된다는 뜻이니까요.

탐색 단계에서는 RTK가 잘 맞습니다. 
파일 구조 파악, 변경 파일 확인, 반복 테스트 결과 훑기 등을 할 때요.

이 단계에서는 원문 전체보다 “어디를 봐야 하는가, 무엇이 바뀌었는가”가 더 중요하니까요.

반대로 정밀 디버깅에서는 원문 로그를 봐야 합니다.

라이브러리 버전 충돌, 비동기 테스트 실패, TypeScript 타입 에러, 빌드 로그 중간에 묻혀 있는 작은 경고까지 직접 확인해야 할 때가 있습니다. 이럴 때 압축된 출력만 믿으면 위험할 수 있습니다.

이런 상황에서는 그냥 Claude에게 명시적으로 말하면 됩니다. “이번에는 RTK 요약 말고 원본 로그 기준으로 봐줘.”라구요. 

—

이 도구를 보면서 다시 생각하게 된 건, 
AI 코딩의 품질이 모델 성능만으로 결정되지 않는다는 점입니다.

무엇을 읽게 할 것인가.
어떤 순서로 보게 할 것인가.
무엇을 컨텍스트에 남길 것인가.
어떤 정보는 요약하고, 어떤 정보는 원문으로 확인할 것인가.

이런 정보 설계가 점점 중요해지고 있는 것 같습니다.

—

AI에게 모든 로그를 다 먹이는 건 
성실한 게 아니라 비효율일 수 있습니다. 
사람도 책상이 너무 어지러우면 판단이 흐려지죠. 
AI에게도 비슷한 일이 일어나는 것 같더라고요.

조금 과장하면, RTK는 토큰 절약 도구라기보다 
AI 에이전트의 작업 환경을 정리해주는 도구에 가깝다는 생각이 들었습니다.

—

코딩을 더 잘하는 것은 AI가 읽는 정보의 품질을 관리하는 일까지가 그 안에 들어가는 게 아닐까요. 무엇을 보여주고, 무엇은 안 보여줄 것인가. 

이걸 의식하기 시작하면 같은 모델을 쓰면서도 결과가 꽤 달라지는 것 같습니다.

저도 아직은 어디까지 압축하고 어디서부터 원문을 봐야 하는지, 그 경계를 가늠하는데 애를 써야하는데요. 

잘하고 싶은 마음은 다르지 않은데, 잘하는 방법은 도구 하나 알아갈 때마다 조금씩 갱신되네요.

—

저는 새로운 도구와 방법을 알려주는 사람이 좋습니다. 제가 보지 못한 것을 볼 수 있게 해주고 자극이 되니까요. 

잘 모르면 현재 아는 것이 전부인양 우쭐하게 될 수 있는데 차라리 그럴 겨를 없이 모르는게 많은 상태이길 바랍니다. 쳐내지 않고 흡수할 수 있게요. 

세상은 너무나 빠르게 달라지고, 때로는 새로 배워야 하는게 버거울 정도로 많게 느껴지는데요.

그래도 압도되지 않고 그저 꾸준히 익혀가는 수 밖에 없는 것 같습니다. 지도교수님과 함께 했던 학술과제명이기도 했던 “학이사 사이학“을 떠올리게 됩니다.
