AI 에이전트를 프로덕션에 올릴 때, 성능보다 중요한 건 '실패 시 복구 능력'이었습니다.

도구를 사용하는 기본 흐름은 단순합니다. 
[AI → 도구 호출 → 결과 수신 → 다음 행동 결정]
하지만 실제 운영 환경에서는 '도구가 실패했을 때' 진짜 실력이 드러납니다.

내부 테스트 중 수많은 실패 케이스를 마주했습니다.
- 도구 에러 시 AI가 같은 호출을 반복하며 비용/지연 급증
- 동일 파라미터 중복 호출로 인한 DB 부하 2배 증가
- 외부 API 마비 시 전체 루프 블로킹 등

저희가 찾은 해법은 '에러의 유형별 자동 피드백 루프' 설계였습니다.
단순히 "에러 발생"이라고 전달하면 AI는 같은 행동을 되풀이합니다. 하지만 에러를 세밀하게 분류해 피드백하면 AI는 스스로 전략을 수정합니다.
- 도구 부재: "다른 접근 경로를 탐색해"
- 타임아웃: "파라미터를 단순화해서 1회만 재시도해"
- 권한 없음: "사용자에게 즉시 안내해"

여기에 Circuit Breaker, Dedup Cache를 조합해 드문 실패가 시스템 전체를 흔들지 않도록 안전장치를 구축했습니다. 이는 업계에서도 공통으로 고민하는 지점입니다.
- LangGraph: v1.0.1에서 도구 에러 처리 기본값 변경 이슈 대응
- MCP: 2026 로드맵에서 Retry 및 Expiry 정책을 핵심 과제로 채택
- SHIELDA 논문: LLM 에이전트 예외 처리를 위한 36개 유형 분류 체계 제시

결국 프로덕션 에이전트의 수준은 "얼마나 똑똑한가?"보다 "실패했을 때 얼마나 안전하게 스스로 복구하는가"에서 결정됩니다.
실제 구현 과정과 상세한 기술적 배경은 블로그에 더 자세히 정리해 두었습니다. (링크는 첫 번째 댓글에 있습니다.)
